# Библиотека ДИИП — сценарии внедрения ИИ

> Часть методологии **ДИИП** (Дизайн ИИ-Преобразований).

Воспроизводимые сценарии применения ИИ в среднем бизнесе. Только реальный опыт — каждая
статья описывает то, что мы или партнёры действительно внедряли.

**Live:** https://scenarios.diip-method.ru/

---

## Таксономия

Дерево организовано по **процессной цепочке создания ценности** (метод ДИИП),
а не по департаментам. Три уровня (масштаб **5 / 29 / 85**):

```
Подсистема (5)  ->  Бизнес-процесс (29)  ->  Подпроцесс (85)  ->  Сценарий
```

URL статьи: `/<subsystem>/<process>/<subprocess>/<slug>/`

Подсистемы (L1): `strategy` (Стратегия и проектирование), `value-stream` (Создание и поставка ценности), `data` (Данные, аналитика и интеллект), `support` (Обеспечение), `management` (Управление и контроль).

Полное дерево (все процессы и подпроцессы с slug’ами) — в [`site/lib/tree.ts`](site/lib/tree.ts).
Узлы показаны все; пустые предлагают «предложить первый сценарий».

## Как добавить сценарий

Добавление статьи = **один MDX-файл + Pull Request**. Править код не нужно.

1. Выберите узел из [`site/lib/tree.ts`](site/lib/tree.ts): `subsystem` -> `process` -> `subprocess`.
2. Создайте файл `site/content/<subsystem>/<process>/<slug>.mdx`
   (имя папки = slug процесса; подпроцесс указывается в frontmatter).
3. Заполните frontmatter:

```yaml
---
title: "Название сценария"   # если есть двоеточие — в кавычках (YAML)
description: Одно предложение — что это даёт бизнесу
author: Имя Фамилия
authorOrg: Диалектика
publishedAt: 2026-07-01
subprocess: <slug подпроцесса из tree.ts>
# process: <slug процесса>   # опционально, если папка не совпадает с процессом
industries: [universal, services, manufacturing]
forWhom: [роль-1, роль-2]
---
```

4. Напишите статью по единому шаблону (ниже).
5. Откройте Pull Request — и дождитесь зелёной сборки (CI). После merge — деплой.

### Формат статьи (единый шаблон)

```
Лид — 1-2 предложения: суть (что делает ИИ). Реальный кейс можно вплести сюда.

## Что болит
Боль клиента + 2-3 блока <Stat before="было" after="стало" />.

## Что делает ИИ
Как работает; при необходимости <Steps> (пайплайн).

## Вариации (настраивается под клиента)   [опционально]
<ToolList><Tool name="..." role="..." /></ToolList>

## Что компания получает
- выгода 1 / выгода 2

## Где нужен человек
<Callout type="warning">Граница ответственности.</Callout>
- честное ограничение 1 / 2
```

Доступные MDX-компоненты: `<Stat>`, `<Steps>`, `<ToolList>`/`<Tool>`, `<Callout>`.

### Принципы контента

- Только реальный опыт — никаких гипотетических сценариев.
- Один сценарий = один узел дерева; кросс-срез — через теги отраслей.
- У каждой статьи есть автор; сценарии партнёров — с указанием команды.

## Структура репозитория

```
ai-scenarios/
\- site/                        # Next.js-приложение
   |- content/                  # Статьи-сценарии (MDX): <subsystem>/<folder>/<slug>.mdx
   |- lib/tree.ts               # Каноническое дерево 5/29/85 (source of truth)
   |- lib/content.ts            # Парсер статей + привязка к узлам
   |- components/               # UI + MDX-компоненты (Stat/Steps/ToolList/Callout)
   \- app/                      # роутинг: главная, [...slug], industries/[tag]
```

> Старые файлы, чьи папки не совпадают с каноническим процессом, привязаны через `NODE_MAP`
> в `content.ts`. Новые статьи NODE_MAP не требуют — только frontmatter.

## Локальная разработка

```bash
cd site
npm install
npm run dev      # http://localhost:3000
npm run build    # статический билд + индекс Pagefind в site/out/
```

Требуется Node.js >= 20.9. Стек: Next.js 16 (`output: 'export'`), MDX, Tailwind CSS v4, Pagefind.

---

## Деплой

Самохостинг на VPS (nginx, статика из `/var/www/ai-scenarios`).
Пересборка и публикация — скриптом `~/redeploy-ai-scenarios.sh`
(git pull -> `npm ci` -> `npm run build` -> чистая публикация `site/out` в web-root).

На каждый PR с изменениями в `site/` работает GitHub Action валидации сборки
(`.github/workflows/ci.yml`, `npm ci && npm run build`) — сломанный MDX не пройдёт.
