# Как добавить сценарий

Эта инструкция самодостаточна: прочитав её, агент или человек может сразу
оформить сценарий правильно. Добавление = **один MDX-файл + Pull Request**.
Править код не нужно.

## TL;DR

1. Прочитай [`site/lib/tree.ts`](site/lib/tree.ts) — выбери узел `subsystem → process → subprocess`.
2. Создай `site/content/<subsystem>/<process>/<slug>.mdx`.
3. Заполни frontmatter (см. ниже) и напиши статью по шаблону.
4. Проверь: `cd site && npm run build`.
5. Открой Pull Request.

## 1. Выбери узел дерева

Дерево — процессная цепочка ДИИП, масштаб 5 / 29 / 85:
**Подсистема → Бизнес-процесс → Подпроцесс**. Источник истины —
[`site/lib/tree.ts`](site/lib/tree.ts). Открой его и выбери подпроцесс,
к которому относится сценарий.

Подсистемы (L1):

| slug | Подсистема |
|------|------------|
| `strategy` | Стратегия и проектирование |
| `value-stream` | Создание и поставка ценности |
| `data` | Данные, аналитика и интеллект |
| `support` | Обеспечение |
| `management` | Управление и контроль |

Процессы (29) и подпроцессы (85) со слугами лежат внутри `tree.ts`
(`SUBSYSTEMS[].processes[].subprocesses[]`). Из дерева тебе нужны три слуга:
`subsystem`, `process` (станет именем папки) и `subprocess` (пойдёт в frontmatter).

## 2. Создай файл

Путь: `site/content/<subsystem>/<process>/<slug>.mdx`

- `<process>` — slug процесса (имя папки).
- `<slug>` — короткий kebab-case на латинице, напр. `lead-qualification`.
- Подпроцесс в путь **не** входит — он указывается в frontmatter.

Пример: подсистема `data`, процесс `analytics`, подпроцесс `reporting` →
файл `site/content/data/analytics/management-reporting.mdx`,
URL `/data/analytics/reporting/management-reporting/`.

## 3. Frontmatter

```yaml
---
title: "Название сценария"        # строка; если есть двоеточие — в кавычках
description: Одно предложение — что это даёт бизнесу
author: Имя Фамилия               # автор обязателен
authorOrg: Диалектика             # компания или команда автора
publishedAt: 2026-07-01           # дата ГГГГ-ММ-ДД
subprocess: reporting             # slug подпроцесса из tree.ts — задаёт место в дереве
# process: analytics              # опционально: указать, если имя папки ≠ slug процесса
industries: [universal, services] # только из списка ниже
forWhom: [руководитель, РОП]      # роли-читатели, свободный список
---
```

### `industries` — допустимые значения

`universal` · `manufacturing` · `services` · `retail` · `it-company` ·
`education` · `healthcare` · `fintech` · `construction`

`universal` — если сценарий не привязан к отрасли. Источник — `INDUSTRIES` в `tree.ts`.

## 4. Структура статьи (единый шаблон)

Строго этот порядок разделов:

1. **Лид** — 1–2 предложения без заголовка: суть, что делает ИИ. Реальный кейс можно вплести сюда.
2. `## Что болит` — боль клиента + 2–3 блока `<Stat>`.
3. `## Что делает ИИ` — как работает; при необходимости `<Steps>`.
4. `## Вариации (настраивается под клиента)` — *опционально* — `<ToolList>`.
5. `## Что компания получает` — маркированный список выгод.
6. `## Где нужен человек` — `<Callout type="warning">` + честные ограничения.
7. Блок «Где внедряли» (`<Callout type="note">`) — реальный кейс в самом конце статьи.

### MDX-компоненты

```mdx
<Stat label="Что меряем" before="было" after="стало" />

<Steps>
1. Шаг один
2. Шаг два
</Steps>

<ToolList>
  <Tool name="Периодичность" role="ежедневно / еженедельно / месяц" />
  <Tool name="Формат" role="таблица / PDF / дайджест" />
</ToolList>

<Callout type="warning">
  Граница ответственности: что остаётся за человеком.
</Callout>
```

Доступны только: `<Stat>`, `<Steps>`, `<ToolList>`/`<Tool>`,
`<Callout type="warning|note">`. Импортировать их не нужно — подключены глобально.

## 5. Полный пример

```mdx
---
title: "ИИ-аналитик: руководитель вовремя видит проблемы в продажах"
description: ИИ собирает данные из CRM и таблиц, считает показатели и сам присылает руководителю отчётность с отклонениями
author: Валентин Демихов
authorOrg: Диалектика
publishedAt: 2026-06-30
subprocess: reporting
industries: [universal, services, retail, manufacturing]
forWhom: [руководитель, РОП, собственник]
---

ИИ собирает данные из CRM и других систем, считает показатели, выделяет отклонения и регулярно присылает руководителю готовую отчётность — компания видит картину до закрытия периода.

## Что болит

Управленческую отчётность собирают вручную: выгрузка из CRM, сверка с телефонией, оформление. Пока собрали — данные уже неоперативны.

<Stat label="Сбор отчётности" before="вручную, 3–6 ч" after="автоматически по расписанию" />
<Stat label="Когда видно проседание" before="после закрытия периода" after="в моменте" />

## Что делает ИИ

По расписанию собирает данные, считает показатели, выделяет отклонения и пишет краткое резюме с выводами.

## Что компания получает

- Руководители работают с актуальными данными, а не с устаревшими таблицами.
- Отклонения видно вовремя — можно исправить до потери выручки.

## Где нужен человек

<Callout type="warning">
  ИИ собирает и анализирует; управленческие решения остаются за руководителем.
</Callout>

- **Качество данных.** Мусор на входе → мусор в отчёте.
- **Интерпретация.** Выводы делает руководитель.

<Callout type="note">
  **Где внедряли.** В розничной сети из 12 магазинов — сбор отчётности по продажам и остаткам из CRM и 1С, ежедневная рассылка руководителю в Telegram. Время на подготовку отчётов сократилось с нескольких часов до нуля.
</Callout>
```

## 6. Правила

- **Только реальный опыт** — никаких гипотетических или выдуманных сценариев.
- Один сценарий = один узел дерева. Кросс-срез по отраслям — тегами `industries`.
- Блок «Где нужен человек» обязателен и честный — без «ИИ решит всё».
- У каждой статьи есть автор.
- Реальный кейс — отдельным блоком внизу. Тело сценария обезличенное; конкретный
  клиент — только в `<Callout type="note">` с жирным зачином «Где внедряли.» в самом
  конце статьи.

## 7. Проверка и PR

```bash
cd site
npm run build   # должно пройти без ошибок
```

Битый frontmatter, несуществующий `subprocess` или сломанный MDX уронят сборку.
Прошло — открывай **Pull Request** и заполни чеклист из шаблона. Мы ревьюим и мёржим.
